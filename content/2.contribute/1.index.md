# Kontribusi

> NLFTs adalah proyek komunitas - dan kami sangat menyukai kontribusi dalam berbagai bentuk! ‚ù§Ô∏è

Ada berbagai cara yang dapat Anda lakukan untuk berkontribusi pada ekosistem NLFTs.

## Ekosistem

Ekosistem NLFTs mencakup banyak proyek dan organisasi yang berbeda:

- [tuxi/](https://github.com/nlfts/tuxi) - repositori inti untuk kerangka kerja NLFTs itu sendiri. [**nlfts/tuxi**](https://github.com/nlfts/tuxi) berisi kerangka kerja Tux (versi 1 Beta tester).
- [nlfts-pixel/](https://github.com/nlfts-pixel) - modul dan pustaka yang dikontribusikan dan dipelihara oleh komunitas. Meskipun modul-modul ini memiliki pemelihara masing-masing, mereka tidak bergantung pada satu orang saja.

## Cara Berkontribusi

### Kelola Isu dan Bantu dalam Diskusi

Periksa isu dan diskusi untuk proyek yang ingin Anda bantu. Sebagai contoh, berikut adalah [papan isu](https://github.com/nlfts/nlfts/issues) dan [diskusi](https://github.com/orgs/NLFTs/discussions) untuk NLFTs. Membantu pengguna lain, berbagi solusi sementara, membuat reproduksi, atau bahkan menyelidiki sedikit bug dan membagikan temuan Anda akan membuat perbedaan besar.

### Membuat Isu

Terima kasih telah meluangkan waktu untuk membuat isu! ‚ù§Ô∏è

- **Melaporkan bug**: Periksa [panduan kami](/contribute/laporkan-bug) untuk beberapa hal yang perlu dilakukan sebelum membuka isu.
- **Permintaan fitur**: Pastikan belum ada isu atau diskusi yang ada yang mencakup ruang lingkup fitur yang Anda pikirkan. Jika fitur tersebut untuk bagian lain dari ekosistem Nuxt (seperti modul), pertimbangkan untuk mengajukan permintaan fitur di sana terlebih dahulu. Jika fitur yang Anda pikirkan bersifat umum atau API-nya belum sepenuhnya jelas, pertimbangkan untuk membuka diskusi di bagian **Ide** untuk berdiskusi dengan komunitas terlebih dahulu.

### Kirim Permintaan Tarik (Pull Request)

Kami sangat mengapresiasi setiap kontribusi berupa pull request! ‚ù§Ô∏è

#### Sebelum Memulai

- **Untuk perbaikan bug**, pastikan sudah ada isu yang membahasnya. Ini membantu memastikan masalah yang Anda temukan bukan hanya masalah dokumentasi atau memerlukan konteks khusus.
- **Untuk fitur baru**, diskusikan terlebih dahulu dengan cara [membuat isu permintaan fitur](/contribute/issue). Hal ini untuk mengonfirmasi apakah fitur tersebut sesuai dengan visi proyek, sehingga dapat mempercepat proses penggabungan.
- **Untuk perubahan besar pada NLFTs**, sangat disarankan untuk [membuat modul](/contribute#buat-modul) terlebih dahulu sebagai bentuk pembuktian konsep. Setelah itu, Anda bisa [membuat RFC](/contribute#buat-rfc) untuk didiskusikan dengan komunitas.
- **Untuk perbaikan kecil** seperti typo, sebaiknya kumpulkan beberapa perubahan dalam satu pull request agar riwayat commit lebih bersih.

#### Konvensi Commit

Kami menggunakan Metode [Conventional Commits](https://www.conventionalcommits.org). Ini membantu kami membuat catatan perubahan (changelog) secara otomatis.

- Gunakan `fix:` atau `feat:` hanya untuk **perubahan kode** yang memengaruhi logika program.
- Gunakan `docs:` untuk perubahan dokumentasi atau `chore:` untuk perubahan kecil lainnya (misalnya, typo).
- Jika bekerja di proyek monorepo (seperti `nlfts/tuxi`), sebutkan lingkup perubahan dalam tanda kurung. Contoh: `feat(kit): menambahkan utilitas 'addMagicStuff'`.

#### Membuat Pull Request

1.  Jika Anda baru pertama kali membuat pull request, pelajari [panduan GitHub ini](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/proposing-changes-to-your-work-with-pull-requests/creating-a-pull-request).
2.  Pastikan judul pull request Anda juga mengikuti [Konvensi Commit](#konvensi-commit) yang telah dijelaskan.
3.  Jika pull request Anda terkait dengan isu tertentu, sebutkan nomor isu tersebut di deskripsi.
4.  Tidak masalah jika pull request Anda memiliki banyak commit. Kami akan menggabungkannya menjadi satu commit saat proses penggabungan nanti (`Squash and Merge`).
5.  Sebelum mengirim, pastikan proses *linting* dan pengujian (*test*) berjalan lancar.
6.  Usahakan pull request hanya berfokus pada **satu perubahan saja**. Jika ada beberapa perubahan yang tidak saling terkait, lebih baik dipisah ke dalam pull request yang berbeda.
7.  Hindari menyertakan perubahan yang tidak relevan (seperti perubahan spasi atau format yang tidak sengaja dilakukan oleh editor).

#### Setelah Pull Request Dikirim

- Kami akan meninjau pull request Anda secepat mungkin.
- Jika pull request ditugaskan ke pemelihara tertentu, artinya orang tersebut yang akan bertanggung jawab meninjaunya.
- Jika kami meminta perubahan, jangan berkecil hati. Itu adalah proses standar untuk memastikan kualitas kode.
- Jika kami menandai PR sebagai 'tertunda', itu adalah catatan internal bahwa kami perlu melakukan tugas lain sebelum meninjau. Kami akan menjelaskan alasannya melalui komentar.
- Kami akan mengikuti [diagram alir keputusan PR](https://mermaid.live/view#pako:eNp9VE1v2kAQ_SsjXzBSEqlALlaUisSh0ACK2l4qcVm8Y9hi7672Iwly-O-ZtYPt5FAOCHbee_PmzdpVlCmOURLlhXrJ9sw4-JNuJNBnWs1UQafIQVjrERyWumAOv58-AJeXt29_0b7BXbWwwL0uRPa1vlZvcB_fF8oiMMmB2QM4BXkt3UoON7Lh3LWaDz2SVkK6QGt7DHvw0CKt5sxCKaQoWQEGtVHcZ04oGdw04LTVngW_LHOeFcURGGz97mw6PSv-iJdsi0UCA4nI7SfNwc3W3JZit3eQ1SZFDlKB15yswQ2MgbOjbYeatY3n8bcr-IWlekYYaJRcyB04I9gOB1CEfkF5dAVTzmFAtnqn4-bUYAiMMmHZgWhNPRhgus5mW2BATxq0NkIZ4Y4NbNjzE2ZchBzcHmGLe_ZMSKCcyRXyLrVFa_5n_PBK2xKy3kk9eOjULUdltk6C8kI-7NFDr8f4EVGDoqlp-wa4sJm3ltIMIuZ_mTQXJyTSkQZtunPqsKxShV9GKdkBYe1fHXjpbcjlvONlO9Kqx_M7YHmOmav_luxfE5zKwVs09hM5DLSupgYDlr5flDkwo7ykixKG-xDsUly1LZ-uY32dgDc7lG7YqwbNp0msJwmIUivjWFtfd-xRrEcJ7Omydz37qFplHOtxEp4GskI2qB5dRCWakglOz3oV8JuITJa4iRL6yZk5bKKNPBGOead-H2UWJc54vIiaW53SPgwrz4fIhVNm1bw76lfI6R2_MW21) dalam meninjau dan merespons setiap pull request.



### Kontribusi Berbantuan AI

Kami menyambut penggunaan alat AI yang bijaksana saat berkontribusi pada NLFTs, namun meminta semua kontributor untuk mengikuti [4 prinsip inti](https://davingm.com/blog/empat-prinsip-inti-kontribusi-dengan-ai)

#### Jangan biarkan LLM berbicara untuk Anda

- Semua komentar, isu, dan deskripsi permintaan tarik harus ditulis dengan gaya bahasa Anda sendiri
- Kami menghargai komunikasi manusia yang jelas daripada tata bahasa atau ejaan yang sempurna
- Hindari menyalin-tempel ringkasan yang dihasilkan AI yang tidak mencerminkan pemahaman Anda sendiri

#### Jangan biarkan LLM berpikir untuk Anda

- Jangan ragu untuk menggunakan alat AI untuk menghasilkan kode atau mengeksplorasi ide
- Hanya kirimkan kontribusi yang Anda pahami sepenuhnya dan dapat jelaskan
- Kontribusi harus mencerminkan penalaran dan pemecahan masalah Anda sendiri

Tujuan kami adalah memastikan kualitas dan mempertahankan kegembiraan dalam berkolaborasi dan berkomunikasi dengan orang sungguhan. Jika Anda memiliki ide untuk meningkatkan kebijakan kami tentang AI di komunitas NLFTs, kami ingin mendengarnya! ‚ù§Ô∏è

### Buat Modul

Jika Anda telah membangun sesuatu yang keren dengan Tuxi, mengapa tidak mengekstraknya menjadi modul?, sehingga dapat dibagikan dengan orang lain? Kami sudah memiliki [banyak modul yang sangat baik](/modules), tapi selalu ada ruang untuk lebih.

Jika Anda membutuhkan bantuan saat membangunnya, jangan ragu untuk [bertanya kepada kami](/contribute/FAQ).


#### Gunakan Core Pack

[Corepack](https://github.com/nodejs/corepack) memastikan Anda menggunakan versi yang benar untuk manajer paket saat Anda menjalankan perintah yang sesuai. Proyek mungkin memiliki bidang `packageManager` di `package.json` mereka.

Di bawah proyek dengan konfigurasi seperti yang ditunjukkan di bawah ini, Corepack akan menginstal `v7.5.0` dari `pnpm` (jika Anda belum memilikinya) dan menggunakannya untuk menjalankan perintah Anda.

```jsonc [package.json]
{
  "packageManager": "pnpm@7.5.0"
}
```


##### Pengaturan IDE

Kami merekomendasikan menggunakan [Anti Gravity](https://antigravity.google/) bersama dengan [ekstensi ESLint](https://marketplace.visualstudio.com/items?itemName=dbaeumer.vscode-eslint). Jika Anda mau, Anda dapat mengaktifkan perbaikan otomatis dan pemformatan saat Anda menyimpan kode yang sedang Anda edit:

```json [settings.json]
{
  "editor.codeActionsOnSave": {
    "source.fixAll": "never",
    "source.fixAll.eslint": "explicit"
  }
}
```

#### Tanpa Prettier

Karena ESLint sudah dikonfigurasi untuk memformat kode, tidak perlu menduplikasi fungsionalitas dengan Prettier. Untuk memformat kode, Anda dapat menjalankan `yarn lint --fix`, `pnpm lint --fix`, `bun run lint --fix`, atau `deno run lint --fix` 

Jika Anda memiliki Prettier terinstal di editor Anda, kami sarankan Anda menonaktifkannya saat mengerjakan proyek untuk menghindari konflik.

#### Manajer Paket

Kami merekomendasikan `pnpm` sebagai manajer paket untuk modul, pustaka, dan aplikasi.

Penting untuk mengaktifkan Corepack untuk memastikan Anda menggunakan versi manajer paket yang sama dengan proyek. Corepack sudah terpasang di versi node baru untuk integrasi manajer paket yang mulus.

Untuk mengaktifkannya, jalankan

```bash [Terminal]
corepack enable
```

Anda hanya perlu melakukan ini satu kali, setelah Node.js diinstal di komputer Anda.

Berikut adalah beberapa tips yang dapat membantu meningkatkan dokumentasi Anda:

- Hindari kata-kata subjektif seperti *cukup*, *hanya*, *tentunya...* jika memungkinkan.<br />

Ingatlah pembaca Anda dapat memiliki latar belakang dan pengalaman yang berbeda. Oleh karena itu, kata-kata ini tidak menyampaikan makna dan dapat berbahaya.
